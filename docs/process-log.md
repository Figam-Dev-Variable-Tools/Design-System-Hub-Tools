# 프로세스 로그 — 병목·사고·개선

여러 에이전트가 병렬로 대규모 리팩터를 돌리면서 관찰된 것들을 기록한다.
**같은 실수를 두 번 하지 않기 위한 문서**이고, 여기서 나온 규칙은 `CLAUDE.md`로 승격된다.

## 🔴 최우선 사고 — 공유 워크트리에서 `git stash`

- **무슨 일**: 한 에이전트가 "검증 게이트의 기준선을 잡겠다"며 `git stash`를 실행했다.
  워크트리를 **모든 에이전트가 공유**하므로, **다른 에이전트 53개 파일의 미커밋 작업이 HEAD로 되돌아갔다.**
- **왜 위험한가**: 되돌아간 파일도 **타입체크는 통과한다**(원본이 유효하니까).
  그래서 아무도 알아채지 못한 채 "완료" 보고가 나가고, 작업은 조용히 사라진다.
  실제로 에이전트들은 자기 편집이 디스크에 있다고 믿고 다음 파일로 넘어갔다.
- **복구**: `stash@{0}`가 살아 있었다. 되돌아간 53개만 골라 `git checkout stash@{0} -- <paths>` 로 복원
  (현재 편집 중인 파일은 건드리지 않는 비파괴 방식). stash는 최종 검증까지 보존.
- **재발 방지 1차 (문서)**: `CLAUDE.md §0-0` — git 상태를 바꾸는 명령 전면 금지.
  기준선이 필요하면 `git show HEAD:<path>` 로 읽거나 파일을 복사한다. 커밋·푸시는 오케스트레이터만 한다.
- **재발 방지 2차 (기계)** ← **문서만으로는 부족했다. 금지라고 적어둔 규칙이 그대로 뚫렸다.**
  `.claude/settings.json` 의 PreToolUse 훅 `.claude/hooks/guard-git.mjs` 이
  **작업 트리를 버리는 명령을 셸에 닿기 전에 exit 2 로 차단**한다. 모든 에이전트·모든 세션에 적용된다.
  - 차단: `stash`(list/show 제외) · `checkout`(`-b` 제외) · `switch`(`-c` 제외) · `restore` · `reset --hard/--merge/--keep`
    · `clean` · `rebase` · `merge` · `revert` · `cherry-pick` · `branch -D/-d` · `worktree remove` · `push --force`
  - 우회로도 막는다: 합성 명령(`cd x && git stash`) · PowerShell 경유 · `bash -c "…"` · `xargs git checkout`
  - 허용: 읽기 전용 git 전부, `add`/`commit`/`push`(비강제), 브랜치 **생성**(`checkout -b`)
  - 오탐 방지: 인용 힙독 안의 **문서 텍스트**("git stash"라고 *적는* 것)는 명령이 아니므로 통과시킨다.
  - **가드 자체를 결함 주입으로 검증한다**: `node .claude/hooks/guard-git.test.mjs` (차단 18 · 허용 15 = 33건).
    게이트를 믿으려면 게이트가 실제로 무는지 증명해야 한다.
- **교훈**: 병렬 에이전트에게는 "파일을 쓰는 권한"만 주고 **"저장소 상태를 바꾸는 권한"은 주지 않는다.**
  그리고 **지키라고 적는 것과 못 어기게 만드는 것은 다르다.** 사고를 낸 규칙은 기계로 강제하라.

## 🔴 두 번째 사고 — 게이트가 초록인데 화면이 깨져 있었다

- **무슨 일**: Figma 세트의 속성 이름을 규약대로 개명하자, 그 세트를 쓰는 **화면의 오버라이드가 조용히 끊겼다.**
  화면 생성기의 `inst(세트, { 속성: 값 })`은 **없는 속성 이름을 경고만 하고 무시**하기 때문이다(warn-and-ignore).
- **피해**:
  - `ProductCard.Price` 개명 → **상품 카드 10장이 전부 세트 기본가로 렌더**(세일가 표기 소실)
  - `SortBar.Count` → **상품 10장짜리 쇼핑 화면이 "6개"를 표시**(HEAD부터 이미 깨져 있던 것)
  - `MemoBox.Counter/Save`, `DropZone.Action` ×3, `SiteHeader/SiteFooter.Brand` …
- **왜 아무도 몰랐나**: **5개 게이트가 전부 초록이었다.** tsc·parity·mapping·naming 어느 것도 이걸 보지 않는다.
  사람이 눈으로 21건을 찾아 고쳤고, **그러고도 6건을 놓쳤다.**
- **대응**: `scripts/verify-screen-props.mjs` 신설 — 모든 `inst()` 속성/축/스왑 이름이 대상 세트에 실재하는지 검사(S1~S5).
  파싱 못 한 호출은 **경고가 아니라 실패**(조용한 통과 금지). 결함 주입 7종 전부 잡는 것을 확인했고,
  복원은 sha256 바이트 동일로 증명했다(git 명령 없이 `cp` 백업으로).
- **핵심 교훈**: **개명은 규약 준수인 동시에 사고다.** `verify-naming`이 초록으로 바뀌는 바로 그 순간 화면이 깨진다.
  → 규칙: **세트 속성을 개명했으면 그 세트를 부르는 화면도 함께 고친다.**
- **파생 교훈**: "고쳐서 통과"와 "안 봐서 통과"는 구별해야 한다. 게이트를 만들 때는
  **결함을 일부러 주입해 실제로 잡는지 확인**해야 한다(그렇게 해서 실제로 6건을 더 잡았다).

## 관찰된 병목 (원인 → 대응)

### B-1. 컨벤션이 문서화되어 있지 않았다 (가장 큰 병목)
- **증상**: 모든 에이전트가 매번 토큰 규칙·강조색 소유권·show 객체 규약을 처음부터 발견했다.
  발견에 실패하면 자기 방식으로 지어냈다(= 할루시네이션).
- **대응**: `CLAUDE.md` 신설(절대 규칙 · 검증 게이트 · 네이밍 파리티 · labels 규약 · 함정 목록).

### B-2. 에이전트가 같은 파일을 반복 탐색했다
- **증상**: 배치마다 10~15분을 "어디에 무엇이 하드코딩됐는지" 찾는 데 썼다.
- **대응**: 감사를 한 번만 돌리고 결과(838건, file:line + 원문 + 제안 타입)를 배치별 JSON으로 잘라 넘겼다.
  → 탐색 단계 제거.

### B-3. 공용 컴포넌트에 축이 없어 각 화면이 우회로를 만들었다
- **증상**: `ListToolbar`에 `totalLabel`·`onSearch`·`disabled`가 없어서
  7개 목록이 제각각 "래퍼 div에서 keydown 가로채기" 같은 우회로를 만들었다.
  결과적으로 "총 24건" 같은 문구가 화면에서 사라졌다.
- **대응**: 우회로를 걷어내고 컴포넌트에 축을 추가. `CLAUDE.md §6`에
  **"우회로를 만들고 싶으면 그건 축이 빠졌다는 신호 — 우회하지 말고 보고하라"** 로 명문화.

### B-4. 셸(AdminListPage)의 기능 공백이 마이그레이션 중간에 발견됐다
- **증상**: 콘텐츠 목록 3개를 옮기려던 에이전트가 `toolbarSelects`·`columnVisibility`·서버 페이징이 없어
  **작업을 못 하고 되돌아왔다**(코드는 안 건드림 — 옳은 판단).
- **대응**: 셸을 먼저 보강하고 재실행. 교훈: **셸을 만들 때 소비자 전수를 먼저 스캔**해야 한다.

### B-5. 순차 의존이라고 착각한 병렬 가능 작업
- **증상**: B1(프리미티브)이 B2~B6의 선행조건이라고 판단해 대기했다.
- **실제**: 의존은 **공용 타입 하나**(`src/shared/labels.ts`)뿐이었다. 그것만 먼저 만들면 전부 병렬 가능.
- **대응**: 공용 타입을 먼저 만들고 B2·B3·B6를 즉시 병렬 착수.

### B-6. 에이전트가 하위 작업을 띄우고 턴을 끝냈다
- **증상**: 부모 에이전트가 자식 6개를 spawn한 뒤 "기다린다"며 종료 → 검증 단계가 실행되지 않았다.
- **대응**: 부모에게 재개 메시지를 보내 검증까지 시켰다.
  교훈: **에이전트에게 "끝까지 하고 보고하라"를 명시**하고, 완료 조건(게이트 통과)을 못 박는다.

## 남은 부채 (오너 판단 필요)

| 부채 | 왜 남았나 | 해소 방법 |
|---|---|---|
| **`variantItem`이 BOOLEAN/TEXT를 문서 상태로 못 켠다** | VARIANT 축만 이름으로 설정 가능. BOOLEAN·TEXT는 전체 키(`showFooter#12:3`)가 필요. 그래서 규약대로 `Card.footer` 축 → `showFooter` BOOLEAN으로 바꾸면 **문서의 '푸터 있는 카드' 그림이 사라진다** → allowlist 3건 | `variantItem`이 `propKeys`로 이름을 풀게 고치면 자동 해소 |
| **변형 폭발 (선재 문제)** | `DS/Button` 432변형(variant6 × appearance3 × size3 × disabled2 × fullWidth2 × iconOnly2). `DS/ImageCard` 72, `DS/TextField` 48 | 축 일부를 BOOLEAN/TEXT로 내리거나 세트를 쪼갠다 |
| **allowlist 중 31건은 "표현 불가"가 아니라 "연기"** | 모양 불변 제약 때문에 미룬 것(AdminTable 툴바·로딩·빈 상태, SearchPanel 접기 토글 등). 사유는 정당하나 **만료가 없어 영구 면제로 굳을 위험** | allowlist에 `until` 필드를 달고 세트 확장 배치를 예약 |
| **`components.ts`가 검증 대상 밖** | 도달 불가(ui.html이 `components:false` 하드코딩)라 당장 위험은 없지만, **검사받지 않는 생성기가 트리에 남아 있다** | 되살리려면 네이밍 검사에 먼저 편입(헤더에 경고 박아둠) |

### 다음에 같은 실수를 막는 방법 (일반화)

1. **사고가 난 규칙은 문서로 끝내지 마라 — 기계로 강제하라.** 훅·게이트·타입 중 하나로 내려라.
   문서는 읽히지 않을 수 있지만 exit 2 는 무시할 수 없다.
2. **게이트를 만들면 결함을 주입해 무는지 증명하라.** "초록"과 "안 봤음"은 구별되지 않는다.
   실제로 5개 게이트가 전부 초록인 채로 상품 카드 10장이 깨져 있었다.
3. **조용한 실패를 만들지 마라.** `inst()` 의 warn-and-ignore, 파서의 "못 읽으면 통과" 가 사고의 근원이었다.
   못 읽었으면 **실패**시켜라.
4. **공유 워크트리에서는 상태를 바꾸는 권한을 나눠주지 마라.** 파일 쓰기 권한과 저장소 상태 변경 권한은 다르다.
